Abstract

When a physically challenged person (blind, deaf or dumb) wants to navigate to a specific location, he/she needs assistance. Our project focuses on helping these people by automating a wheelchair based on the user’s input and requirements. Difficulties for a physically challenged person like navigating paths can be overcome by using a GPS enabled computer. Inputs can be given to the computer through voice or an in-built keyboard. Possible paths and querries are displayed on the screen and also read out by the computer. The smallest distance path will be set as the default path but can be changed by the user. Our project uses a Beagle Bone Black, GPS, a wifi dongle and a wheelchair with motors and a battery. When the computer is turned on, the current location is read out and it waits for the destination to be input by the user. The user is indicated by voice, and by using the display as such. These can be customized by users based on their requirements.
Inputs can be given to the computer using the Android keyboard or through voice which can be activated by speaking out a keyword.
Once the location has been fed, the location is displayed on LCD display and the destination is spoken out. After confirmation, all possible routes are computed and displayed on the LCD.
Upon another confirmation, the wheelchair moves along the path set, and upon reaching a new location the computer reads the location name and proceeds to the destination.
The wheelchair will have Ultrasonic sensors at the front and rear to prevent collisions. Alerts can be invoked through speakers and the display.
After reaching the destination the wheelchair gives a voice command signal indicating that destination reached successfully.

A. Hardware Implementation

Our project has a test robot for prototype and real time model is a wheel chair capable of carrying one person. The test robot has four wheels, uses 100rpm motors with differential drive for taking turns. It uses a 12V battery 8 AH for power supply and is controlled by Beagle Bone Black. The wheel chair is powered from a 12V 25AH car battery and uses two 12v high torque power window motors to drive 28 INCH wheels and TI’s DRV8412 motor driver to control motors. These motors are capable of 100rpm and deliver a maximum torque of 9Nm each. The wheelchair uses gear drive for transmission. The steering of the wheel chair is controlled by a 60 rpm DC motor. This 60 rpm motor uses TI’s L293D motor driver for control. The maximum speed this wheel chair can attain is 9Kmph. The implementation also includes 4 ultrasonic sensors for obstacle detection (preventing collision).

 

B. Software Implementation

The software was written in Java and XML. An Android application was developed and was interfaced with our embedded system. Provided the origin and destination, the app accesses Google maps and gets the data regarding the path to be travelled from origin to destination and sends the processed data to the embedded system. The embedded system and the Android mobile can be connected either by WiFi or by WiFi P2P (Peer to peer).
After the splash screen with a welcome message, the app prompts the user to say one of the following two words to select the mode of input,

  • Voice
  • Touch
    When the input mode is voice the user has to say the destination and press the journey button and the journey starts.
    The app has a sliding drawer called options and inside which are three tabs,
  • WiFi tab
  • Map settings tab
  • Route tab
    The WiFi tab has a button to connect the mobile phone with the embedded system. The WiFi tab also has buttons to move the vehicle forward (‘F’), backward (‘B’), left (‘L’), right (‘R’) or to stop (‘S’) the vehicle.
    The map setting tab has the option to select one of the following views for the map,
  • Normal view
  • Terrain view
  • Satellite view
  • Hybrid view
    The app accesses the magnetic field sensors of the android device and the orientation of the mobile phone with respect to the Earth North is displayed.
    When the user selects touch input, the origin and destination should be specified in the Route tab. The user can select the origin and destination in one of the following ways,
  • Text mode
  • Latitude Longitude mode
  • Marker mode
  • Text mode asks the user to specify the input in the form of a string. Latitude and Longitude mode asks the user to specify the origin and destination as latitude and longitude. Marker mode spawns two markers; one marker for origin and another for destination; into the Google maps.
  • The origin marker is fixed to the current location if the checkbox, “Current location as origin is checked”, otherwise it is movable. The user can reposition the origin and destination markers as required.
  • Now, to get the route, the user should press the ‘R!’ button. The app access Google servers via a HTTP request and the Google servers respond with a HTTP response. The HTTP request is sent via an URL encoding.
    The response can be obtained in one of the two formats,
  • JSON (Javascript Object Notation)
  • XML (Extensible Markup Language)
    The format of the response can be specifies in the URL when the request is made in the first place.
    Examples of HTTP requests,
    http://maps.googleapis.com/maps/api/directions/json?origin=Tanjore&destination=Trichy
    http://maps.googleapis.com/maps/api/directions/xml?origin=Tanjore&destination=Trichy
    Now the data obtained is parsed and the route is overlaid on the Google Maps.
    The user should press the ‘J!’ button to prepare the whole system for the journey. Now a violet marker is displayed on the next check point to which the user must travel.
    The user must press the ‘JOURNEY’ button inside the WiFi tab to start the journey. The current location(latitude and longitude) of the vehicle, the current angle of the vehicle with respect to Earth’s North, the distance between the current location and the next checkpoint and the total journey distance to be travelled is displayed on the right upper portion of the user interface.
    The current orientation of the vehicle, the target orientation of the vehicle and the distance to be travelled is transmitted to the embedded system from the Android mobile through WiFi.
    Initial testing was done using a raspberry pi and then we moved on to a much more powerful board, Texas Instrument’s BeagleBone Black. The embedded system gets this data and uses it to guide the vehicle. As the current location of the vehicle changes and the vehicle enters the green circle beneath the violet marker, the current checkpoint is reached. Now the violet marker and the green circle beneath the violet marker move towards the next checkpoint. As the vehicle moves, it passes through all the checkpoints and at some point of time the checkpoint to be reached will be same as the destination marker. When the destination is reached the vehicle stops and sends an alert to the user, ‘MISSION ACCOMPLISHED’.
    The clear (‘C’) button can be used to get rid of all the objects that were overlaid on the Google maps.
